TestPlan: Github Rate-limited backoff for 403/429

Skip to end of metadata
  • Created by Sudhamini Tirumalagari, last modified about 5 hours ago
Go to start of metadata
  1. OF-39784:

    Handle tenant level rate limited backoff when any tenant gets rate limited with 403/429 for contents/users api

    Issue: The rate limited events for spec based csps are not being pushed to delayed exchange queue as they're not getting classified as retryable exceptions.
    Due to this, we're not able to get the retry after milli secconds and the retry event is not picking up the retry after millis from the csp, but using the default value of 20ms.
    Also the event is not being pushed to the delayed exchange queue at the event handler as the exception is a runtime exception and not a CloudserviceRateLimit/Retry exception.

    Fix: The rate limited events for spec based csps are being pushed to delayed exchange queue as they're classified as retryable exceptions.
    enable:use-delayed-exchange

OF-39644:Retry_timer is not honouring when github csp hits RateLimit with 403/429 for contents/users api 

Issue: retry_after_milliseconds is not honouring when github csp hits RateLimit with 403/429 for contents/users api 

Fix: retry_after_milliseconds is honouring when github csp hits RateLimit with 403/429 for contents/users api ,

TestPlan:

Scenario1:When github contents api gets ratelimted
FFs enabled:
       use-delayed-exchange
       use-proxy-for-dlp-auto


TestSteps:

  1. set below expectation from mockserver to get 429 response for contents api

    {
  "httpRequest": {
    "method": "GET",
    "path": "\/repos\/skyhightest4org1\/.*\/contents\/.*"
    // This matches the path: /repos/skyhightest4org1/{any_repo}/contents/{any_path}
  },
  "httpResponse": {
    "statusCode": 429,
    "headers": {
      "Content-Type": ["application/json; charset=utf-8"],
      "Retry-After": ["600"] 
      // Instructs the client to wait 600 seconds before retrying
    },
    "body": {
      "type": "JSON",
      "json": {
        "message": "You have exceeded a secondary rate limit for the organization skyhightest4org1. Please retry your request after 600 seconds.",
        "documentation_url": "https://docs.github.com/rest/overview/resources-in-the-rest-api#secondary-rate-limits"
      }
    }
  },
}
  2. create or upload a file to the git repo and verify that 429 error received for the contents api call

  3. once file is sucessfully uploaded then webhook received the event then sending it to zeus event processor

  4. after that  "\/repos\/skyhightest4org1\/.*\/contents\/.*" api hitting then returning 429 response
    expected behaviour:

  5. In redis Zeus queue key gets created :offlinedlp:103170:21423:30891:ratelimited 

  6. TTL of the above key should be the set as Retry-After of expectations (step1)(by default it will be 20ms)
  7. Meanwhile new events not getting processed when retry_after_milliseconds timer has started  and  that events are being added to delayed exchange queue
  8. Once the key (offlinedlp:103170:21423:30891:ratelimited) TTL gets expired then the key will be  deleted then this event and new events will be processed
    9. redis encriched events behaviour.

Scenario2:When github users api gets ratelimted
  FFs enabled at instance level:

       use-delayed-exchange
       use-proxy-for-dlp-auto
TestSteps:

  1. set below expectation from mockserver to get 429 response for contents api

    {
"httpRequest": {
"method": "GET",
"path": "\/users\/.*"
},
"httpResponse": {
"statusCode": 429,
"headers": {
"Content-Type": ["application/json; charset=utf-8"],
"Retry-After": ["600"] 
// Instructs the client to wait 60 seconds before retrying
},
"body": {
"type": "JSON",
"json": {
"message": "You have exceeded a secondary rate limit for the organization skyhightest4org1. Please retry your request after 60 seconds.",
"documentation_url": "https://docs.github.com/rest/overview/resources-in-the-rest-api#secondary-rate-limits"
}
}
},
"times": {
"remainingTimes": 1,
"unlimited": false
},
"timeToLive": {
"unlimited": true
}
}
  2. create or upload a file to the git repo and verify that 429 error received for the users api call
  3. once file is successfully uploaded then webhook received the event and then event will send to redis and stored under "offlinedlp:csp_spec_event:tenant:csp:instance: key.
  4. after that  "\/users\/."  GET api hitting then returning 429 response
    expected behaviour:

  5. In redis Zeus queue key gets created :offlinedlp:103170:21423:30891:ratelimited 

  6. TTL of the above key should be the set as Retry-After of expectations (step1)
  7. Meanwhile new events not getting processed when retry_after_milliseconds timer has started  and  that events are being added to back to redis
  8. Once the key (offlinedlp:103170:21423:30891:ratelimited) TTL gets expired then the key will be  deleted then this event and new events will be moved to rabbit_MQ

    Scenario3:

    Validate the behaviour for other spec based csps which have rate limit handling capability
